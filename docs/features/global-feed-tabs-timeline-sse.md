# Глобальная лента: вкладки, хронология, SSE (план реализации)

**Статус:** в реализации (MVP: `GET /api/feed/global`, SSE `/api/feed/global/events`, фронт вкладки Всё/Посты/Карточки, FAB ленты, локальные просмотры; legacy `GET /api/cards/feed` с `mode` пока сохранён).

## Цель

- Убрать выбор режима рекомендаций («Для вас» / «Из подписок» / «От подписчиков»).
- Три вкладки с **одним смыслом «публичное приложение»**: **Всё** (посты + карточки, вкладка по умолчанию), **Посты**, **Карточки** — отличаются только типом контента.
- Лента на сервере: **предсказуемая глобальная хронология** по времени публикации/создания (единый ключ сортировки для карточек и постов), **без** опоры на соцграф в MVP.
- Локально: **«смотрел»** только после **открытия детальной страницы** поста или карточки; визуальное отличие просмотренного — на клиенте.
- Индикатор **«есть новое сверху»**: **SSE**; при свёрнутом приложении / обрыве сети уведомление может **запаздывать** до следующего подключения (**без** head-polling при фокусе в MVP).
- Кнопка «вверх» (см. существующую [`ScrollToTopFab`](../../frontend/src/components/navigation/ScrollToTopFab.tsx)): **красная точка** при новой активности сверху; после скролла наверх — смена на **reload** и явное **обновление данных** + обновление локального **`last_feed_refresh_at`**.

## Зафиксированные решения (MVP)

| Тема | Решение |
|------|--------|
| Источник контента | Глобальный публичный поток; **без** фильтра «только мой круг» в MVP. |
| Вкладки | **Всё** (default) / **Посты** / **Карточки** — один пул, фильтр по типу. |
| Старые `mode` | Не использовать в UI и API нового потока (legacy можно удалить или оставить за флагом — на этапе реализации). |
| «Смотрел» | Только **деталка**; хранение **локально** (как в Reddit в браузере). |
| Новизна сверху | **SSE**; **без** отдельной страховки head-polling при фокусе. |
| Бэкенд-форма | **Один** таймлайн-эндпоинт + keyset cursor `(sort_at, id)`; вкладка = параметр `kind`. |
| Сортировка на сервере | Строго по времени; **без** серверного буста «не открывал». |

## Технический подход

### API ленты

- Ввести (или заменить текущий) **`GET`** с параметрами: `kind=all|posts|cards`, `limit`, опционально `cursor` (keyset).
- Ответ: дискриминированный список элементов (карточка / пост), **`next_cursor`**, при необходимости **`feed_version`** или **`head_sort_at`** для согласования с SSE.
- Правила **публичности** — строго по модели БД (черновики, приватные профили и т.д. — явный список в задаче реализации после аудита схемы).
- Убрать из ответа зависимость от **`feed_source`** для рекомендаций или заменить одним значением вроде `global` для ясности контракта.

### SSE

- Отдельный маршрут **SSE** с той же авторизацией, что и API.
- Минимальное событие: **изменилась голова** глобальной ленты (например монотонный `version` или `max(sort_at)`), без тела списка.
- Клиент: подписка **только на экране ленты** (или на всех трёх вкладках — уточнить при реализации жизненный цикл соединения); при событии и `scrollY` ниже порога — показать **красную точку** на FAB.

### Фронтенд

- Заменить сегменты `FEED_MODE_ENTRIES` на три вкладки по **`kind`**.
- Три набора query-ключей / один хук с параметром `kind`; кэш и восстановление скролла — **по вкладке** (`kind`, не старый `mode`).
- Локальное хранилище: множество **просмотренных id** (пост / карточка) при успешном заходе в деталку; **`last_feed_refresh_at`** при успешном refresh после reload наверху.
- Визуально: опциональная подсветка «не открывал» на основе локального набора (не менять порядок с сервера или только мягкие маркеры — зафиксировать в PR).

### Бэкенд

- Новый сервис или сильное упрощение [`list_movie_card_feed`](../../backend/src/services/cards/list_movie_card_feed.py): вместо потоков `subscriptions` / `affinity` / … — **UNION-подобная** выборка публичных карточек и постов, `ORDER BY sort_at DESC, id DESC`, keyset pagination.
- Публикация SSE: при создании публичной карточки / поста — **уведомить** подписчиков канала (in-memory / Redis pub-sub — выбрать по текущей инфраструктуре проекта).
- Тесты: контракт API (пустая лента, только посты, только карточки, курсор, публичность), SSE smoke (подключение + одно событие), регрессии старых маршрутов если остаются.

## План работ (порядок)

1. **Аудит публичности** — какие поля у `MovieCard` / `FeedPost` и `User` определяют попадание в глобальную ленту; edge cases (удалённые, забаненные — если есть).
2. **Контракт API** — OpenAPI/схемы Pydantic, `kind` + cursor; задепрекейтить или удалить `mode` на `/api/cards/feed` (решение в PR: один новый путь vs замена существующего).
3. **Реализация выборки** — один SQL/ORM-путь, индексы под `(sort_at, id)` для карточек и постов.
4. **SSE** — эндпоинт, источник событий при insert, лимиты и reconnect на клиенте.
5. **Фронт** — вкладки, React Query, FAB с точкой и reload, локальные viewed + `last_feed_refresh_at`.
6. **Тесты и ручная проверка** в Mini App (фон, обрыв SSE, смена вкладки).

## Ограничения MVP (явно для пользователей и поддержки)

- «Новое сверху» **не гарантировано** мгновенно без активного SSE.
- «Смотрел» **не синхронизируется** между устройствами.
- **Нет** фильтра по подпискам; глобальная лента может потребовать **антиспам/модерацию** позже.

## Следующие итерации (не MVP)

- Серверный waterline / синк «просмотрено» между устройствами.
- Опциональный фильтр «только мой круг».
- Страховка при фокусе: лёгкий `HEAD`/`peek` если отказались от strict-only-SSE.
