# Любимые карточки в ленте и рекомендации

Отдельная продуктово-техническая фича: **как отмеченные автором любимые карточки живут в ленте**, что с ними делает UI (помимо текущего тихого оформления постера), и **какой сигнал они несут для ранжирования и смешивания** в рекомендательной системе.

Связанные документы: `docs/features/feed-recommendation-engine.md`, `docs/features/movie-card-favorites.md`.  
Текущая реализация UI-акцента на постере: `frontend/src/components/feed/FeedAuthorFavoritePosterChrome.tsx`, использование в `FeedCard`.

## Что уже сделано (MVP)

- В ответе ленты у карточки есть признак любимого у автора (`is_favorite` с бэкенда).
- В ленте для таких карточек применяется **нелитеральное** оформление постера: виньетка, внутреннее мятное свечение, лёгкий grain, скобы «видоискатель», усиленное кольцо/тень у ссылки на постер. Без текста «любимое» и без сердца **в чужой ленте** (сердце остаётся только у владельца для управления).

## Что нужно продумать дальше

### 1. Контракт API и поле для фронта

- Нужен ли отдельный флаг уровня ленты, например `feed_highlight: 'author_favorite' | null`, чтобы не привязывать визуал только к `is_favorite` и иметь место для других типов выделения?
- Должен ли признак «любимое автора» быть **виден подписчикам и в discovery** одинаково, или скрываться в части сценариев (приватность, A/B)?

### 2. Поведение в ленте (не только внешний вид)

- Ограничение **частоты**: не показывать подряд много «акцентных» карточек от одного автора.
- Согласованность с **источником** (`feed_source`): нужен ли микро-буст позиции для любимых из подписок vs discovery?
- **Анти-спойлер для смысла**: визуал не раскрывает словами «любимое», но для части пользователей стоит ли добавить опциональное объяснение в настройках («так выглядят особо отмеченные автором фильмы»)?

### 3. Рекомендательная система

Имеет смысл явно зафиксировать в `feed-recommendation-engine.md` (или здесь расширить и потом слить):

| Тема | Вопросы |
|------|---------|
| **Вес сигнала** | Насколько «favorite» сильнее обычной высокой оценки? Нужен ли отдельный вес в графе user–film? |
| **Ограничение числа любимых** | Если у пользователя много любимых, не размывается ли сигнал? Нужен лимит в продукте или saturation в скоринге? |
| **Смешивание слотов** | Участвуют ли любимые в `SLOT_PATTERN` / fallback отдельно (например, редкий слот «сильный сигнал от подписок»)? |
| **Коллаборативка** | Использовать пересечения множеств любимых для похожих пользователей при разреженных оценках? |
| **Разнообразие** | Штраф за избыток любимых одного автора в окне из K карточек. |
| **Discovery** | Как любимые влияют на показ контента вне соцграфа (только как вторичный признак похожести жанров/тегов или как явный буст)? |
| **Эксперименты** | Метрики: CTR на постер, удержание, жалобы «не понимаю почему карточка другая» — нужен ли качественный опрос. |

### 4. Тестирование и наблюдаемость

- Логировать ли в аналитике показ карточки с типом `author_favorite_spotlight` (без PII)?
- Нужны ли snapshot-тесты / визуальные тесты на наличие слоёв оформления (хрупко) или достаточно e2e на классы/data-attr?

### 5. Документация

- После утверждения правил ранжирования — обновить `docs/features/feed-recommendation-engine.md` секцией «Сигнал author_favorite».
- Кратко описать UX-язык выделения в `docs/features/feed-ui-card-design.md` (принципы без текста и иконки «любимое»).

## Статус

- **UI-акцент постера**: реализован (см. компонент выше).
- **Изменения ранжирования ленты под любимые**: не внедрялись; требуют отдельного технического решения и, при необходимости, эксперимента.
