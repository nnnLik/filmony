# Feature Request

## Metadata
- Feature slug: `telegram-engagement-notifications`
- Author: Codex
- Created at: 2026-05-07 09:00 UTC
- Priority: high
- Target area: backend-heavy (+ frontend для «поделиться» и deep link)
- Related draft: `.cursor/features/009-telegram-notifications.md` (общая инфраструктура бота/Celery)

## Problem
- Пользователь хочет **направленно** отправить созданную карточку одному из тех, на кого он **подписан**, и чтобы адресат получил **простое сообщение в боте** (что случилось, от кого, ссылка в Mini App).
- Пользователь должен получать уведомление в боте, когда кто-то ставит **эмоцию/реакцию** на его **карточку** или **комментарий** (когда доменная модель реакций будет доступна).
- Сейчас в репозитории **нет** исходящей доставки через Bot API и **нет** модели реакций в backend — фича должна заложить контракт событий и MVP без «раздувания» текста в чате.

## Scope
- In scope:
  - Событие **«поделился карточкой с подписчиком»** (отправитель выбирает получателя только из своих **подписок** `following`).
  - Подготовка к событиям **реакция на карточку** и **реакция на комментарий** (уведомление **владельцу** сущности: автор карточки / автор комментария).
  - Минимальный текст сообщения в Telegram: **тип события**, **от кого** (отображаемое имя), **одна ссылка** в Mini App на нужный экран.
  - Идемпотентность / анти-спам окно, rate limits, аудит (outbox или лог).
  - Валидация: нельзя слать себе, нельзя слать не-подписке, учёт блокировок если появятся в модели.
- Out of scope (v1):
  - Rich-карточки Telegram, картинки постера в сообщении, кнопки с несколькими действиями.
  - Экран настроек «категории уведомлений» в приложении (можно заложить API-заготовку).
  - Дайджесты и группировка «N человек отреагировали» (см. улучшения ниже).

## Concrete Requirements
- [ ] После создания карточки (или с экрана карточки) пользователь может вызвать API **«отправить карточку пользователю U»**; `U` обязан быть в списке **моих подписок**; иначе `403` или `422` с явной причиной.
- [ ] При успешном запросе получатель с валидным `telegram_user_id` получает **одно** сообщение от бота с шаблоном: действие + имя отправителя + ссылка (deep link в Mini App на карточку).
- [ ] При появлении доменных событий реакции: **владелец карточки** получает уведомление о реакции на карточку; **автор комментария** — о реакции на свой комментарий; **не уведомлять**, если реакцию поставил сам автор сущности.
- [ ] Дедупликация: повторная отправка «та же карточка → тот же получатель» в окне **X минут/часов** не порождает второе сообщение (или возвращает идемпотентный `200` без повторной доставки — зафиксировать в API).
- [ ] Rate limit: лимит на число исходящих **share** и **reaction-notify** с одного пользователя в единицу времени (конфиг).
- [ ] Ошибки Bot API (`403` chat not found, user blocked bot) не должны ломать основной user flow создания карточки/реакции; фиксировать статус доставки для поддержки.
- [ ] Ссылка в Mini App должна открывать **конкретную карточку** (и при необходимости якорь комментария — v1.1).

## Success Criteria
- [ ] В staging/dev тестовое событие share приводит к **ровно одному** сообщению у адресата при соблюдении лимитов.
- [ ] Неверный получатель (не подписка) не вызывает отправку в Telegram.
- [ ] Повторные клики/запросы не создают заметного спама за счёт dedupe + rate limit.
- [ ] Deep link из сообщения открывает нужную сущность в Mini App (после доработки роутинга по `start_param`, если ещё не готово).
- [ ] Секреты и токены бота не попадают в логи; ретраи без дублирования пользовательских сообщений.

## MVP (Minimal First Version)
- [ ] **Только share карточки** подписчику + инфраструктура: Bot API client, очередь (Celery или эквивалент), таблица/outbox попыток доставки.
- [ ] Один шаблон текста RU для share; реакции — **заглушка события** или отложены до появления таблицы реакций.
- [ ] Dedupe ключ: `(sender_id, recipient_id, card_id, event_type)` с TTL.
- [ ] API: например `POST /api/cards/{card_id}/share` с телом `{ "recipient_user_id": ... }`.
- [ ] Frontend: выбор получателя из списка подписок + вызов API (можно вторым PR после backend).

## Что легко упустить (закрываем в спеке)
- **Получатель должен иметь возможность получить личное сообщение от бота**: пользователь хотя бы раз нажал **Start** в чате с ботом (иначе `chat not found` / нет `chat_id`). Нужно явное поведение: тихий fail + опционально подсказка отправителю «пользователь не запускал бота».
- **Связь user ↔ telegram**: у нас пользователь создаётся из Mini App; для DM нужен стабильный идентификатор чата. Зафиксировать, откуда берём `chat_id` (обычно `telegram_user_id` из профиля при условии взаимодействия с ботом).
- **Нет обработчика updates бота в проде** — см. `README.md`; для исходящих `sendMessage` webhook не обязателен, но для `/start` и будущих кнопок может понадобиться.
- **Часовые пояса и тексты**: короткий шаблон без лишних полей; локаль v2.
- **GDPR / отписка**: минимум — команда бота «стоп» или настройки позже; в MVP — не рассылать при флаге `notifications_enabled=false` если добавим поле.

## Улучшения и оптимизация (после MVP)
- **Группировка**: «Иван и ещё 3 человека отреагировали на ваш комментарий» раз в N минут.
- **Приоритет очереди**: share выше фоновых реакций или наоборот — по продукту.
- **Настройки**: выключить «реакции», оставить «шары»; mute пользователя.
- **Batch** в Redis: счётчик реакций за окно, одно сообщение.
- **Метрики**: delivery rate, причины fail, spam reports.

## Constraints
- Technical constraints:
  - Доставка не в HTTP-request критического пути: ставить задачу в worker, API отвечает быстро.
  - Один публичный entrypoint на сервис уровня «отправить уведомление по событию» или отдельные сервисы по одному действию — по стандартам репозитория.
  - Тесты: mock Bot API; интеграционный тест на валидацию share без реального Telegram.
- Product constraints:
  - Сообщения **минималистичные**: без markdown-излишеств, одна ссылка.
  - Share только на **тех, на кого я подписан** (как в твоём описании).

## Acceptance Criteria
- [ ] Share карточки неподписчику невозможен на уровне API.
- [ ] Успешный share ставит задачу доставки; получатель видит сообщение при валидном бот-чате.
- [ ] Дубликаты и burst-спам режутся dedupe + rate limit.
