# Feature Request

## Metadata
- Feature slug: `feed-recommendation-engine`
- Author: Codex
- Created at: 2026-05-06 20:53 UTC
- Priority: high
- Target area: backend

## Problem
- Лента должна учитывать социальный граф (подписки/подписчики) и личную релевантность (оценки/активность), но без однообразия.
- Нужно контролируемо добавлять контент от пользователей вне графа, чтобы "разбавлять" поток.
- Выдача должна быть быстрой, стабильной между запросами и предсказуемой при пагинации.

## Scope
- In scope:
  - Архитектура источников ленты и их веса.
  - Алгоритм смешивания: подписки, подписчики, персональный сигнал, discovery-слой.
  - Правила стабильной пагинации и обновления ленты.
  - Базовый anti-spam/anti-dup набор для ленты.
  - Контракты API выдачи ленты.
- Out of scope:
  - Сложные ML-модели и офлайн обучение.
  - Полноценный real-time push слой нотификаций (можно предусмотреть совместимость).

## Concrete Requirements
- [ ] Выдача учитывает минимум 4 источника кандидатов: `subscriptions`, `subscribers`, `personal_affinity`, `discovery`.
- [ ] В итоговой странице присутствует не только граф-контент: минимум 1 элемент discovery на каждые 5-10 элементов выдачи (конфигурируемо).
- [ ] Для пользователя не показываются дубликаты карточек в пределах одной сессии пагинации.
- [ ] Порядок выдачи стабилен в рамках одного pagination cursor/snapshot.
- [ ] API поддерживает cursor-based пагинацию и инкрементальную догрузку.
- [ ] На уровне ранжирования учитываются свежесть, релевантность интересам и базовые social signals.
- [ ] Контент с низким качеством (спам/повторы) понижается или отбрасывается.

## Success Criteria
- [ ] p95 ответа feed API укладывается в целевой SLA (например, <= 250-350ms при warm cache).
- [ ] При повторном запросе с тем же cursor пользователь получает тот же стабильный порядок.
- [ ] Доля discovery-контента соблюдается в заданном диапазоне.
- [ ] CTR/engagement discovery не ниже минимального целевого порога (фиксируется продуктово).
- [ ] Нет заметных "пустых" лент для пользователя с малым social graph.

## MVP (Minimal First Version)
- [ ] Кандидатогенерация из `subscriptions` + `subscribers` + простой `discovery pool`.
- [ ] Легковесный scoring без ML: свежесть + социальная близость + совпадение по тегам/жанрам.
- [ ] Детеминированное merge-смешивание с фиксированными квотами источников.
- [ ] Cursor pagination с защитой от дублей.
- [ ] Базовые метрики (latency, source-mix, duplicates) и простые feature-флаги/конфиги.

## Constraints
- Technical constraints:
  - Бизнес-логика в backend должна быть вынесена в отдельный service-класс с одним публичным entrypoint.
  - Производительность приоритетна: избегать N+1 и лишних запросов на каждый элемент ленты.
  - Тесты `pytest` для API и сервиса обязательны (happy path + edge cases).
- Product constraints:
  - Лента не должна превращаться в полностью "случайную": discovery дозируется.
  - Поведение должно быть понятным и объяснимым для продуктовой настройки.
