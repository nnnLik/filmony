# Feature Request

## Metadata
- Feature slug: `movie-card-custom-reactions`
- Author: Codex
- Created at: 2026-05-07 12:00 UTC
- Priority: high
- Target area: fullstack
- Replaces scope: `.cursor/features/006-comments-and-likes.md` (лайки / дефолтные emoji не используются)

## Problem
- Нужны **реакции на карточки и комментарии** без стандартного набора emoji мессенджеров.
- Реакции **кастомные**: админ/продукт задаёт **набор картинок** (ассеты), они хранятся в системе; пользователь выбирает только из этого каталога.
- Должно работать **на чужих и своих** карточках и **на чужих и своих** комментариях (политика self-react — см. ниже).

## Scope
- In scope:
  - Каталог реакций: запись в БД (например `id`, `label` опционально, `image_url` или ключ в object storage, `sort_order`, `is_active`).
  - API: список доступных реакций для клиента; установка/снятие реакции пользователя на цель `movie_card` или `movie_card_comment`.
  - Агрегаты: число реакций по типу на цели; опционально «моя текущая реакция» в ответах карточки/комментария.
  - UI: пикер из каталога; отображение счётчиков и выбранных иконок на карточке и под комментарием.
- Out of scope (v1):
  - Произвольная загрузка картинок конечными пользователями.
  - Нативные Telegram custom emoji как единственный источник (можно позже слить с тем же каталогом, если понадобится).

## Concrete Requirements
- [ ] Нет дефолтного unicode-emoji набора: только реакции из таблицы каталога.
- [ ] Одна активная реакция пользователя на цель: либо **один тип на пользователя** (замена при выборе другой), либо **несколько разных типов** — зафиксировать в плане реализации (рекомендация MVP: **одна реакция на пользователя на цель**, повторный клик по той же — снять).
- [ ] Цели: `movie_card_id` и `comment_id` (комментарий привязан к карточке; валидация существования).
- [ ] Разрешены реакции на **свою** карточку/комментарий и на **чужие** (если продукт запретит self-react — добавить флаг в сервисе).
- [ ] Админское наполнение каталога: минимум ручной SQL/скрипт или внутренний endpoint под защитой (v1 можно без UI админки).

## Success Criteria
- [ ] Пользователь видит только картинки из каталога; выбор сохраняется и отображается другим пользователям.
- [ ] Нагрузка на списки карточек/комментариев остаётся приемлемой (без N+1 на реакции; при необходимости денормализация счётчиков или один запрос агрегатов).
- [ ] Повторное действие идемпотентно и предсказуемо (toggle).

## MVP
- [ ] Таблица каталога + таблица `user_reactions` (user_id, reaction_type_id, target_type, target_id, unique constraint).
- [ ] `GET /api/reactions/catalog`, `POST /api/reactions` (или scoped под card/comment), тесты.
- [ ] Минимальный UI на карточке и в блоке комментариев.

## Constraints
- Секреты и URL хранилища — через конфиг; картинки не хранить в репозитории без необходимости.
- Сервисная логика — отдельные service-классы по стандартам репозитория.

## Acceptance Criteria
- [ ] Реакция на карточку и на комментарий проходят happy path и права доступа (аутентификация).
- [ ] Нельзя применить `reaction_type_id`, отсутствующий в каталоге или выключенный.
