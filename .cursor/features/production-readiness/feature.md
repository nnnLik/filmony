# Feature: production readiness (prod launch)

## Metadata
- **Feature slug:** `production-readiness`
- **Priority:** P0 (блокер публичного запуска)
- **Target area:** infra + fullstack (конфиг, деплой, проверки)
- **Каноничные спеки в репо:** [`.cursor/tech.md`](../../tech.md), корневой [`README.md`](../../../README.md), **[`docker-compose.prod.yml`](../../../docker-compose.prod.yml)** / `vars/.env.production`, [`docs/features/telegram-user-base.md`](../../../docs/features/telegram-user-base.md), [`docs/features/celery-redis-workers.md`](../../../docs/features/celery-redis-workers.md)

## Problem

Выпустить Filmony в продакшен так, чтобы Mini App, API, БД, фоновые уведомления и медиа реакций работали предсказуемо, без утечки секретов и с приемлемой безопасностью.

## Scope

- **In scope:** хостинг, TLS, прокси, env, миграции, сборка фронта, BotFather, smoke-тесты, минимальная эксплуатация (логи, бэкапы).
- **Out of scope:** отдельная фича «webhook /start у бота», ML, полный SRE-контур (если не требуется сразу — вынести в follow-up).

## Success criteria

- [ ] Пользователь открывает Mini App по прод-URL, проходит `POST /api/auth/telegram`, видит ленту и основные сценарии.
- [ ] Шаринг / engagement-уведомления доходят при запущенном Celery worker и Redis.
- [ ] Секреты не в репозитории; прод-`DATABASE_URL` и JWT отделены от dev.

## Чеклист по ходу событий (выполняем по порядку)

Считайте каждый блок **гейтом**: не переходите к следующему, пока не отмечены критичные пункты текущего (если нет осознанного исключения).

### Этап 0 — зафиксировать целевой контур
- [ ] Определён хостер/VPS (или PaaS), регион, объём RAM/CPU под Postgres + app + Redis + worker.
- [ ] Решено: фронт отдаётся с того же домена что API, или раздельно (влияет на CORS и cookie).
- [ ] Выбран домен/поддомен под Mini App (HTTPS обязателен для прод TMA и безопасной cookie-сессии).

### Этап 1 — домен и HTTPS
- [ ] DNS A/AAAA записи указывают на сервер (или на CDN/балансер по вашей схеме).
- [ ] Выпущен и автоматически продлевается TLS-сертификат (например Let’s Encrypt).
- [ ] HTTP → HTTPS редирект включён (если используете Nginx/Caddy).

### Этап 2 — сеть и безопасность хоста
- [ ] Открыты наружу только нужные порты (минимум `443`, опционально ограниченный SSH).
- [ ] Postgres и Redis **не** торчат в публичный интернет без необходимости (лучше только internal/docker network).
- [ ] Запланированы или включены автоматические обновления ОС/образов (политика на усмотрение).

### Этап 3 — данные: PostgreSQL
- [ ] Поднят прод-инстанс PostgreSQL (отдельно от dev).
- [ ] Настроены **бэкапы** (расписание, хранение, тест восстановления хотя бы один раз).
- [ ] Задан надёжный пароль/роль; `DATABASE_URL` хранится в секретах окружения.

### Этап 4 — Redis и Celery
- [ ] Redis доступен сервису бэкенда и **celery-worker** (как в `docker-compose.prod.yml`, или эквивалент в оркестраторе).
- [ ] Worker запускается с тем же образом/кодом и env, что и API (один токен бота, одна `DATABASE_URL`).
- [ ] Проверено: задача в очереди обрабатывается (логи worker без постоянных ошибок подключения).

### Этап 5 — объектное хранилище (если реакции в проде)
- [ ] RustFS/S3-совместимое хранилище поднято, ключи в env (`RUSTFS_*` или ваши имена).
- [ ] При необходимости выполнен синк стикеров/ассетов по вашему `Makefile`/скриптам.

### Этап 6 — секреты и конфиг приложения
- [ ] `TG_APP_TOKEN` совпадает с ботом BotFather (тот же, что для Mini App).
- [ ] `AUTH_JWT_SECRET` — уникальный длинный секрет продакшена, не равный dev.
- [ ] `KINOPOISK_API_KEY` задан.
- [ ] `TELEGRAM_BOT_USERNAME` задан (ссылки «открыть бота»).
- [ ] `ENV`/режим продакшена отключает отладочные вещи (по практикам проекта).
- [ ] Все значения подставляются из секретов/файла env **вне** git (ориентир: `vars/.env.example`).

### Этап 7 — миграции
- [ ] На прод-БД выполнено: `alembic upgrade head` (или ваша обёртка `make migrate` в контексте прод-окружения).
- [ ] Версия схемы совпадает с ожидаемой веткой релиза.

### Этап 8 — сборка и выкат артефактов
- [ ] `npm run build` во `frontend/`; статика отдаётся хостингом фронта (Workers/Pages и т.д.).
- [ ] `docker compose -f docker-compose.prod.yml build && docker compose -f docker-compose.prod.yml up -d`; секреты — в **`vars/.env.production`**.
- [ ] Снаружи: TLS-прокси → порт бэкенда (например **8100**).

### Этап 9 — Telegram / BotFather
- [ ] У бота в BotFather указан **HTTPS** URL Mini App (Menu Button / Web App).
- [ ] Команда понимает: пользователь попадает в БД при открытии Mini App и `POST /api/auth/telegram`, не только при `/start` в чате (см. `telegram-user-base.md`).
- [ ] Для личных сообщений бота получатель хотя бы раз нажал **Start** у бота (ограничение Bot API).

### Этап 10 — смоук-тест перед открытием аудитории
- [ ] Открытие Mini App из Telegram → сессия создана, `GET /api/me` успешен.
- [ ] Лента `GET /api/cards/feed` открывается без 5xx.
- [ ] Создание карточки (resolve film → create card) успешно.
- [ ] Комментарий и реакция (если включены) без ошибок.
- [ ] `POST /api/me/notifications/ping` или шаринг карточки приводит к сообщению в боте при живом worker (проверить хотя бы один сценарий доставки).

### Этап 11 — наблюдаемость и эксплуатация
- [ ] Логи приложения и worker доступны и читаемы (ротация или внешний сбор).
- [ ] Есть способ узнать о падении сервиса (uptime-мониторинг минимум на HTTPS endpoint).
- [ ] Зафиксирован короткий **runbook**: кто перезапускает compose, где секреты, как откатить образ.

### Этап 12 — продукт и право (если публичный запуск)
- [ ] Опубликована или подготовлена **политика конфиденциальности** / обработка данных Telegram-пользователей (по требованиям площадки и юрисдикции).

### Этап 13 — go-live
- [ ] Решение о включении трафика / анонса зафиксировано.
- [ ] После включения — повторный короткий смоук из Telegram клиента (не только curl).

## Follow-up (не блокирует минимальный прод, но полезно рано)

- [ ] Webhook бота для `/start` и онбординга в Mini App.
- [ ] Sentry или аналог для ошибок API/фронта.
- [ ] Rate limiting на публичные эндпоинты при росте трафика.
- [ ] Отдельный документ в `docs/features/production-readiness.md` после выполнения чеклиста (итог, использованные URL, отличия от compose-dev).

## Constraints

- Docker-first для проверок бэкенда в разработке остаётся; прод может повторять паттерн compose или свой оркестратор — зафиксировать одно.
- Cookie-сессия и Telegram `initData` требуют согласованного HTTPS и домена.
