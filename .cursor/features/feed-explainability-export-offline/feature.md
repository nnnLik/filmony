# Feature Request: объяснимость ленты, экспорт карточек, офлайн-кэш ленты

## Metadata
- **Feature slug:** `feed-explainability-export-offline`
- **Priority:** P1 (UX/доверие + владение данными + устойчивость в WebView)
- **Target area:** fullstack (объяснимость и экспорт — backend + контракты; кэш — преимущественно frontend)
- **Depends on:** `feed-recommendation-engine`, `movie-card-*` / карточки с рейтингом и тегами; авторизация по сессии
- **Related:** [`.cursor/user-story.md`](../../user-story.md) (прозрачность рекомендаций), [`docs/features/feed-recommendation-engine.md`](../../../docs/features/feed-recommendation-engine.md)

## Problem
1. Пользователь не понимает, **почему** в ленте показана конкретная карточка — снижается доверие к рекомендациям и соцслою.
2. Нет способа **выгрузить** свои просмотры и оценки для бэкапа, анализа или переноса (CSV / JSON).
3. В Telegram Mini App сеть нестабильна — повторная загрузка ленты при каждом открытии даёт пустой экран и фрустрацию; нужен **устойчивый кэш** последней успешной выдачи на клиенте.

## Scope

### A. Объяснимость в ленте (feed explainability)
**In scope**
- Для каждого элемента ленты (или для доли элементов в MVP — по решению в плане) отображать **короткую человекочитаемую причину** в терминах продукта, согласованную с источниками из [`feed-recommendation-engine`](../../feed-recommendation-engine/feature.md): например «Из подписок», «Вашим подписчикам», «Похоже на ваши теги», «Рекомендация для вас», «Новое в сервисе» и т.п.
- Источник правды — **бэкенд**: в ответе `GET /api/cards/feed` (или вложенно в каждый item) поле вида `feed_reason_code` + опционально `feed_reason_label` (или один локализуемый ключ), чтобы не дублировать бизнес-логику на клиенте.
- Маппинг код → строка UI на фронте (i18n-ready).

**Out of scope (v1)**
- Длинные «объяснения с ИИ», проценты уверенности, A/B без флагов.

**Acceptance (A)**
- [ ] В ответе ленты у элементов, для которых источник определён в merge-слое, присутствует стабильное поле(я), по которым UI показывает подпись без дополнительных запросов на карточку.
- [ ] Подпись не раскрывает приватные данные других пользователей сверх уже видимого в карточке.
- [ ] Документирован контракт и перечень кодов (таблица в `docs/features/` после реализации).

---

### B. Экспорт просмотренных фильмов (CSV и JSON)
**In scope**
- Эндпоинт(ы) для **текущего пользователя**: выгрузка **всех** его карточек фильмов (`movie_card`), по смыслу «уже оценено / зафиксировано в сервисе», с полями, которые реально доступны в модели: минимум связка с фильмом (идентификаторы KP/внутренние), **рейтинг**, теги (структурированные + кастомные), даты, при необходимости slug фильма/название — перечень фиксируется в плане реализации.
- Форматы:
  - **JSON** — массив объектов, стабильные ключи, UTF-8.
  - **CSV** — одна строка заголовка, разделитель запятая или `;` (зафиксировать; экранирование полей с переносами), UTF-8 с BOM опционально для Excel.
- Аутентификация обязательна; rate limit / разумный max размер ответа или потоковая отдача при росте данных (если объём велик — отдельное решение в плане).

**Out of scope (v1)**
- Экспорт чужих профилей, публичные без авторизации.
- XLSX, PDF.

**Acceptance (B)**
- [ ] Авторизованный пользователь может получить полный список своих карточек в JSON одним запросом (или пагинация, если иначе нельзя — явно в плане).
- [ ] То же для CSV с предсказуемой схемой колонок.
- [ ] `pytest` на контракт, пустой список, большая выборка (при необходимости фикстуры).
- [ ] В UI Mini App точка входа «Экспорт данных» (минимум ссылка/кнопка, скачивание файла или копирование — по плану).

---

### C. Офлайн-кэш ленты на клиенте
**In scope**
- После успешного `GET /api/cards/feed` (первая страница или N элементов — порог в плане) сохранять **снимок** ответа в **долговременном клиентском хранилище** (предпочтительно IndexedDB или аналог под React), с метаданными: время сохранения, cursor первой страницы.
- При открытии ленты без сети или до ответа сервера показывать **последний успешный** снимок с явной пометкой «данные от …» / «нет сети» (копирайт по UX).
- Инвалидация: при успешном свежем ответе — обновить кэш; опционально TTL (например сутки) — в плане.
- Не ломать существующую пагинацию: кэш только для первой «страницы» или согласованного объёма.

**Out of scope (v1)**
- Полный offline-first CRUD карточек, фоновая синхронизация всего профиля.
- Service Worker и HTTP-кэш API, если не нужны для TMA — по решению в плане.

**Acceptance (C)**
- [ ] При обрыве сети пользователь видит последнюю загруженную ленту (или пустое состояние с понятным сообщением, если кэша никогда не было).
- [ ] Нет утечки сессии в кэше чужого пользователя на том же устройстве (кэш ключовать по user id или сбрасывать при logout / смене сессии).
- [ ] Тесты на уровне фронта — где покрытие уже принято в проекте (или ручной чеклист в `result.md`, если тестов нет).

## Success Criteria (сквозные)
- Пользователь понимает происхождение хотя бы части элементов ленты по подписи.
- Пользователь может выгрузить свои оценки и метаданные карточек в **CSV** и **JSON**.
- Лента остаётся читаемой при плохой сети за счёт последнего снимка на клиенте.

## Constraints
- Docker-first для backend-тестов ([`.cursor/tech.md`](../../tech.md)).
- Тонкий API-слой, бизнес-логика в сервисах; экспорт — отдельный сервис с одним публичным entrypoint.
- Telegram Mini App: учитывать ограничения WebView (квоты storage, нет фоновых процессов как в нативе).
